02
Anatomy of a Git Commit — 4 Data Layers
Layer 1

Metadata — The Envelope

Identity, timing, and verification data attached to every commit.

  • author.nameWho wrote the code
  • author.emailAuthor's email identity
  • committer.nameWho actually pushed it (can differ)
  • committer.emailCommitter's email identity
  • author.dateTimestamp when code was written
  • verificationGPG signature verified true/false
  • messageDeveloper's commit description
Layer 2

Statistics — The Size

Aggregate change metrics revealing the scope of modifications.

  • stats.additionsTotal lines of code added
  • stats.deletionsTotal lines of code deleted
  • stats.totalCombined change count
  • files_changedNumber of files modified

These stats feed the behavioral baseline engine to detect abnormal commit sizes — a developer who normally changes 30 lines suddenly pushing 2,000+ lines triggers investigation.

Layer 3

Files Changed — The Surface

Every file touched in the commit with its path and change type.

  • filenameFull path: src/config/database.py
  • statusadded / modified / removed / renamed
  • additionsLines added in this specific file
  • deletionsLines deleted in this specific file
  • previous_filenameOriginal name if file was renamed

File paths are matched against sensitive file patterns — touching .env, *.pem, or terraform.tfstate immediately elevates risk.

Layer 4

The Diff — The Actual Code

Line-by-line unified diff showing exactly what was added or removed.

# context line (unchanged)
+ AWS_SECRET = "AKIA3EXAMPLE..."
- access_control = "strict"
+ access_control = "permissive"
# more context

This is the most critical layer. We scan every added line (+) against 58 insider threat SAST rules — detecting backdoors, exfiltration code, secrets, and security control tampering.

4. Author / Committer Identity Mismatch
Source: Layer 1 — Metadata  |  Identity Comparison
+1.5
Risk Points
Git stores two separate identities: the author (who wrote the code) and the committer (who applied it to the repo). When these don't match, it could indicate impersonation, credential sharing, or an attempt to attribute malicious changes to another developer.
What We Compare
author.email vs committer.email

• Match → Normal — same person wrote and committed
• Mismatch → Investigate — someone else applied this code

Note: Legitimate in open-source (maintainer merging PRs), but suspicious in corporate repositories where developers push their own code.
Threat Scenarios
• Credential sharing: Developer A uses Developer B's credentials to push code, distributing blame
• Impersonation: Attacker with repo access attributes malicious commits to a trusted developer
• Compromised account: External attacker pushes code through a compromised developer account while the real developer is offline
Medium Signal
5. Off-Hours Commit Activity
Source: Layer 1 — Metadata + Behavioral Baseline
+1.0
Risk Points
Detects commits made outside the developer's normal working hours. The system learns each developer's typical commit window and flags deviations using z-score analysis — not a fixed time range, but personalized to each individual's pattern.
How Baseline Detection Works
From the last 60 commits, the engine calculates:

• Mean commit hour — e.g., 14:00 (2 PM)
• Standard deviation — e.g., 2.5 hours
• Normal window — mean ± 1.5σ = 10:00 - 18:00

Z-score ≥ 2.0 → Medium anomaly
Z-score ≥ 3.0 → High anomaly

Example: Developer who always commits 9am-5pm suddenly pushes code at 3:12 AM → z-score = 4.5 → High severity timing anomaly.
Why This Matters
• Data exfiltration: Insider pushes code at 3 AM when no one is monitoring
• Compromised credentials: Attacker in different timezone uses stolen credentials
• Pre-departure sabotage: Employee making unauthorized changes before resignation takes effect

Important: A single off-hours commit is low concern. Combined with sensitive file access or large deletions, it becomes a strong compound signal.
Medium Signal
6. Large Code Deletions
Source: Layer 2 — Statistics + Behavioral Baseline
+1.0
Risk Points
Flags commits with unusually high deletion counts — both against an absolute threshold (500+ lines) and relative to the developer's historical average (5x above baseline). Mass deletion is a classic sabotage indicator.
Detection Thresholds
Absolute: deletions > 500 lines → +1.0 points

Behavioral: deletions > 5× developer's baseline average
• 5-10× average → Medium severity
• > 10× average → High severity

Example: Developer averages 25 deletions/commit. A commit with 800 deletions = 32× above baseline → High severity anomaly.
Threat Scenarios
• Sabotage: Deleting critical business logic, infrastructure code, or deployment configs before leaving
• Cover tracks: Removing logging/audit code to hide evidence of prior malicious activity
• IP theft preparation: Removing code from repo after copying it to personal storage
• Denial of service: Deleting database migration files to corrupt deployments
Medium Signal
7. Unsigned Commits (Missing GPG Signature)
Source: Layer 1 — Metadata  |  Verification Field
+0.5
Risk Points
In organizations that mandate commit signing, an unsigned commit indicates the code was pushed without cryptographic identity verification. The developer (or attacker) did not have the private GPG/SSH key needed to sign.
Why Unsigned Commits Are Suspicious
• Compromised credentials: An attacker has the password or PAT token but not the GPG signing key — the commit works but is unsigned
• Policy bypass: Developer intentionally disabling signing to avoid audit trail
• Automation injection: Malicious CI/CD script pushing code without human signing

Context matters: This is a supporting signal, not conclusive on its own. But unsigned + off-hours + sensitive file = strong compound indicator.
Critical Signal
8. Force Push (History Rewrite)
Source: Webhook Event  |  forced: true
+2.0
Risk Points
A force push (git push --force) rewrites repository history, replacing existing commits with different ones. This can permanently erase evidence of previous changes — making it a powerful tool for covering tracks.
What Happens in a Force Push
Normal push: adds new commits on top of existing history
Force push: replaces existing commits — the old code is gone

This means an insider can:
• Push malicious code → wait for it to deploy → force push to erase the evidence
• Remove audit logs that were previously committed
• Rewrite commit authorship to blame others
Detection Method
GitHub/GitLab webhook payload includes "forced": true field.

When detected:
• Compare old ref vs new ref to identify what was removed
• Log the overwritten commits before they're garbage collected
• Alert immediately — force pushes to protected branches are almost always policy violations
Medium Signal
9. Suspicious Commit Messages
Source: Layer 1 — Metadata  |  Message Content Analysis
+0.5
Risk Points
Flags commits with vague, empty, or deceptive messages when the commit contains substantial changes. A developer pushing 500+ lines with the message "fix" or "." is either careless or deliberately hiding intent.
What We Detect
Only triggers on non-trivial commits (>50 lines changed):

• Empty messages — blank or <4 characters
• Single-word vague — "fix", "update", "wip", "test", "tmp", "stuff", "done", "."

Why it matters: Legitimate developers describe their changes. An insider hiding a backdoor in a 300-line commit labeled "misc" is hoping no one looks closer.
Example
commit a4f8b2c
Author: dev@company.com
Message: "."
Changes: +487 -12 across 8 files

→ 487 lines added with a single-character message
→ Suspicious message signal triggered

13. Configuration & Permission Weakening
Source: Layer 4 — Diff on Config Files  |  Security Posture Analysis
+1.0
Per Finding (Cap +2.0)
Detects changes that weaken the application's security posture — broadening CORS policies, disabling MFA, removing rate limits, weakening TLS, enabling debug mode, removing CODEOWNERS protections, and un-ignoring sensitive files in .gitignore.
Weakening Patterns Detected
PatternWhat It Means
CORS wildcard (*)Any website can make authenticated requests to your API
MFA disabledMulti-factor authentication turned off in config
Auth requirement removedrequire_auth = false or equivalent
Rate limit disabledAPI brute-force protection removed
Debug mode enabledStack traces, verbose errors exposed to users
SSL verify disabledHTTPS certificate validation turned off
Weak TLS versionDowngrade to TLS 1.0/1.1 or SSLv3
.gitignore unignoredRemoving .env or .pem from ignore = secrets get tracked
CODEOWNERS removedRemoving required reviewers for sensitive code paths
Why This Is Dangerous
Config weakening is the quietest form of insider threat. No new code is added, no backdoor is inserted — just a small setting change that opens a door:

• Remove yourself from CODEOWNERS → future malicious PRs won't require your review
• Disable MFA → compromised password alone grants access
• Remove .env from .gitignore → secrets start appearing in git history
• Set CORS to * → any malicious website can steal user sessions

These changes are often dismissed as "dev convenience" but can enable devastating follow-up attacks.
